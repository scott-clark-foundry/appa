Skills and Tools subsystem

Context

Three seams in the architecture have always pointed at each other: touching any one drags the other two along.

The scaffold carries three architecture seams that are documented in docs/architecture.md but unimplemented in code: ToolCapability (tool-result handling plus the trust_class/side_effect_class metadata convention), the SkillRegistry and Skill tool (open Agent Skills progressive disclosure), and the persona/core-blocks split of the Instructions composer. tools.py holds empty stubs; the Instructions composer exists and is wired (core_agent.instructions(compose_instructions)) but has zero registered contributors.

These three seams cannot land independently: they form a dependency chain. Skills register as tools, so they require the tool-metadata convention. The skills catalog is an Instructions contributor, so it requires the composer to actually own all Instructions content. Today it does not: config.instruction_text reaches the model through Pydantic AI's static instructions= parameter, which bypasses the composer entirely. The composer captures only contributor output into RunState.captured_instructions, so as long as instruction_text rides the static slot, Run.instructions is incomplete: it is currently the empty string while the model actually receives instruction_text, a violation of the architecture invariant that Run.instructions is "what the model actually saw." Adding the catalog as a contributor while leaving instruction_text static would only widen that gap.

Why one slice

This spec defines the slice that lands all three together, with the Instructions-composer migration as a first-class part of the work rather than a side prerequisite.

Goals

  1. G1
    ToolCapability, a Pydantic AI AbstractCapability, reads trust_class and side_effect_class from each tool's ToolDefinition.metadata and acts on them at the before_tool_execute and after_tool_execute boundaries: verified by unit tests over both branches of each field.
  2. G2
    Every registered function tool declares both metadata fields at definition time; a tool missing either field causes a fail-fast error at agent construction, verified by a test that registers an under-specified tool and asserts the error.
  3. G3
    SkillRegistry discovers SKILL.md files under the project and user skill directories, parses their frontmatter, and renders a catalog string: verified by a test fixture tree containing valid skills, a name collision, and a malformed file.
  4. G4
    A Skill(name) tool, with name constrained to an enum of discovered skill names, is registered when at least one skill is discovered and absent otherwise; invoking it returns the frontmatter-stripped, structurally-wrapped skill body as a tool result.
  5. G5
    AgentConfig.instruction_text is replaced by instructions_frame_text and persona_text; both are registered as Instructions contributors; the static instructions= slot is unused; Run.instructions equals the full composed string the model received, verified by an assertion over a completed Run.
  6. G6
    Each Instructions contributor emits at most one XML-delimited block: an empty contributor (e.g. the skills catalog with no skills discovered) contributes nothing and is skipped: and the scaffold-frame block is registered first, verified by a test asserting the frame block leads and that every non-empty block carries its named tag.
  7. G7
    The <untrusted_content> wrapping path is exercised by a test-only untrusted tool fixture, verified by asserting the wrapper markers appear in the result the model receives.
  8. G8
    An anti-revert content-hash test pins the default instructions_frame_text, consistent with Anti-revert content-hash discipline.

Non-goals

  1. NG1
    The human-in-the-loop approval queue: at L1 the side-effect gate auto-approves (a no-op pass-through). The seam exists; the queue is deferred to L3 per docs/architecture.md ToolCapability.
  2. NG2
    Tier-3 skill resources (scripts/, references/, assets/ directories inside a skill). This slice implements catalog (tier 1) and body (tier 2) only: deferred to a later skills spec.
  3. NG3
    MemoryStore and memory tools: a neighboring deferred subsystem. This spec leaves the memory-index contributor slot vacant (it registers after persona when that subsystem lands).
  4. NG4
    The end-of-run tool-pair lifecycle classifier: skill activations remain ordinary tool-call/tool-return pairs in messages[] carried by normal history persistence. Deferred; revisited only on trace evidence.
  5. NG5
    Non-skill function tools: Skill is the only function tool authored in this slice. The metadata convention is the seam future tools follow; retrofitting is what it deliberately avoids.
  6. NG6
    Source-grounding / provenance metadata on tool results: ToolCapability's documented "(later)" third concern. Deferred.
  7. NG7
    IdentityProvisioner: a separate unrelated seam; not touched here.

Design

ToolCapability

An AbstractCapability subclass registered on the core agent via Agent(capabilities=[…]), alongside the existing Hooks capability. It is the single convergence point for side-effect gating and prompt-injection defense.

  • before_tool_execute: reads side_effect_class from tool_def.metadata. read and write tools pass through unchanged. external and irreversible tools route through the gate. At L1 the gate auto-approves: it returns the validated args unchanged and is observable as a no-op. The gate is a named, exercised code path so that L3 can swap in an approval queue without re-shaping callers.
  • after_tool_execute: reads trust_class from tool_def.metadata. A trusted result passes through unchanged. An untrusted result is wrapped in <untrusted_content> … </untrusted_content> markers before it returns to the model.
Authoring convention

Every registered function tool declares two fields on its ToolDefinition.metadata dict at definition time:

  • trust_class: "trusted" | "untrusted": whether the tool's returned content may carry injected instructions.
  • side_effect_class: "read" | "write" | "external" | "irreversible": read/write auto-execute; external/irreversible route through the gate.

Both fields are required. A tool that registers without both is a configuration error surfaced as early as possible: at agent construction, not at first tool call. Fail-fast is deliberate: a silently un-classified tool is a security hole, not a degraded feature.

This convention depends on Pydantic AI surfacing the metadata where the capability needs it: Tool.metadata is readable at agent construction (the fail-fast check) and ToolDefinition.metadata is readable inside the before_tool_execute / after_tool_execute hooks (gating and wrapping): see docs/pydantic-ai/hooks-reference.md.

SkillRegistry

A startup-time discovery component holding an in-memory registry of skill records.

  • Discovery: scans <project>/.agents/skills/*/SKILL.md then ~/.agents/skills/*/SKILL.md. On a name collision the project-scoped skill wins; the registry warns and continues rather than failing. .claude/skills/ paths are intentionally not scanned: they hold skills for the development agent building this project, not for the runtime agent the scaffold ships: scanning them would leak the builder's tooling into the product agent's catalog.
  • Frontmatter contract: each SKILL.md carries YAML frontmatter. Required: name (≤64 chars, lowercase letters and hyphens, matching the parent directory name) and description (≤1024 chars). Optional fields (license, compatibility, metadata, allowed-tools) are tolerated and ignored in this slice. Parsing uses a lenient fallback so a minor frontmatter defect degrades one skill rather than aborting discovery; a skill whose required fields cannot be resolved is skipped with a warning.
  • Skill record: name, description, and filesystem location. The skill body is not part of the record's contract: it reaches the model only through the Skill(name) tool result, never through the composed instructions. Run.instructions completeness: invariant I2: is therefore independent of whether bodies are read eagerly at discovery or lazily on activation. Either is acceptable; the choice is the implementer's, as bodies are small.

A skill record looks like:

scaffold/skills/registry.pypython
@dataclass(frozen=True)
class SkillRecord:
    name: str          # validated: ≤64 chars, lowercase + hyphens, matches parent dir
    description: str   # validated: ≤1024 chars
    location: Path     # the SKILL.md file path

Skill tool

  • Registered on the core agent only when at least one skill is discovered: an enum with zero members is not a valid parameter type, and a no-op tool is noise in the model's tool list.
  • The name parameter is constrained to an enum of discovered skill names. This prevents the model from hallucinating an activation for a skill that does not exist.
  • Invoking Skill(name) returns the skill's body with YAML frontmatter stripped, structurally wrapped, as a tool_result: the dedicated-tool activation pattern from the Agent Skills standard, chosen over raw file-read so frontmatter stripping and structural wrapping are guaranteed. The wrapper delimits the activated skill content with a named marker:
<skill name="pdf-processing">
…frontmatter-stripped body…
</skill>

This is a structural marker (it tells the model where the skill body begins and ends); it is distinct from the <untrusted_content> marker, which is the injection-defense convention for untrusted results. Skill bodies are trusted.

Metadata: trust_class="trusted" (the body comes from our own filesystem) and side_effect_class="read".

Instructions contributors and the composer migration

AgentConfig.instruction_text is replaced by two fields:

scaffold/config.pypython
class AgentConfig(BaseModel):
    instructions_frame_text: str   # scaffold role contract + output discipline; scaffold default, config-overridable
    persona_text: str              # who the agent IS: name, voice, character; deployment-owned, scaffold default
    # … existing fields unchanged …

Contributor shape. Every Instructions contributor is a (ctx) -> str callable registered with the composer. Each contributor's output is one XML-delimited block: a named tag wrapping a markdown body, so the model locates a section by its tag rather than by its position.

<ROLE>
You are a **master** software engineer. …
</ROLE>

Two helpers keep this uniform:

  • wrap(tag, text): a pure formatting primitive producing <TAG>\n{text}\n</TAG>. Every contributor calls it; it owns no state and takes no ctx.
  • static_block(tag, text): a thin closure factory for blocks whose text is a literal; it returns a (ctx) -> wrap(tag, text) contributor. Used for the frame and persona blocks.

A contributor is a hand-written function only when its text is computed. The skills catalog is the one such contributor in this slice; the future memory index is another. Frame and persona are not bespoke functions: they are two static_block(...) calls.

The composed stack

  1. ROLE Frame: scaffold role contract + output discipline. Scaffold-owned, config-overridable. Names the Skill tool and <untrusted_content> conventions.
  2. PERSONA Persona: who the agent is. Deployment-owned. Becomes a computed contributor reading MemoryStore.core_blocks() when memory lands.
  3. MEMORY Memory index: deferred (NG3). Slot reserved.
  4. SKILLS Skills catalog: one entry per discovered skill. Computed. Empty contributor when no skills discovered; composer skips empty blocks without a stray separator.

The three contributors create_agent() registers:

  1. Frame: static_block("ROLE", config.instructions_frame_text). The scaffold's role contract and output discipline. Its default text additionally names two conventions the model must understand: that it activates a listed skill by calling Skill(name), and that content delimited by <untrusted_content> markers is data, never instructions. Scaffold-owned, config-overridable.
    Known wrinkle, deferred

    The skill-activation sentence lives in the always-present frame, so a deployment that discovers zero skills gets a frame naming the Skill tool when no such tool is registered and no <SKILLS> block exists. It is inert: the sentence is conditioned on "a skill listed in <SKILLS>", and a model can only call tools in its schema: but the cleaner placement is the catalog contributor's output. Fold this move into the next deliberate frame-text edit, since it changes the literal and trips the G8 hash.

  2. Persona: static_block("PERSONA", config.persona_text). Who the agent IS. When MemoryStore lands, this is the one contributor that changes: it becomes a computed contributor reading MemoryStore.core_blocks(), with no change to the composer or to any other contributor.
  3. Skills catalog: a computed contributor. Renders one entry per discovered skill as - {name}: {truncated description}, wraps the result with wrap("SKILLS", …), and returns the empty string when no skills are discovered.
Truncation rule

Search the first 250 characters of description for a sentence terminator (., !, or ?) followed by whitespace or end-of-string; if one is found and the text up to it is ≤200 characters, cut there. Otherwise, if the whole description is ≤200 characters, keep it. Otherwise hard-cut at 200, strip trailing whitespace, append ….

The core agent is constructed with the static instructions= parameter unused: the composer owns all Instructions content. After this change Run.instructions equals the full string the model received.

Registration order

Contributors apply in registration order (the composer is FIFO). Because each block is XML-tagged, the sequence of persona, catalog, and the future memory index is not semantically load-bearing: the model finds <SKILLS> by its tag wherever it sits.

One ordering constraint survives: the frame is registered first, because it is the interpretive contract: it tells the model how to read <untrusted_content> and what Skill(name) does, and the model should read the rule before the content it governs.

The registration order is otherwise fixed so the composed string is byte-stable (the prompt-cache prefix discipline) and is pinned by a contributor-order assertion in the scaffold test suite: a reorder is a visible, reviewed change caught by that assertion, not a semantic error. It is not caught by the anti-revert content-hash test: that test hashes each contributor's text (the instructions_frame_text literal, per G8), which a reorder leaves byte-identical.

This slice also updates docs/architecture.md Instructions composer to match: its rule-3 worked example ("the Skills catalog must precede consumer text") is removed, and the section gains the XML-delimited-block convention and the softened ordering rule.

Invariants

  1. I1
    Every registered function tool has both trust_class and side_effect_class in its ToolDefinition.metadata. No tool reaches a model request un-classified.
  2. I2
    Run.instructions for a successful run equals the byte-for-byte string the model received as its instructions: the composer owns all Instructions content, nothing rides the static slot.
  3. I3
    A skill name exposed in the Skill tool enum corresponds to exactly one discovered skill record (collisions resolved at discovery, project wins).
  4. I4
    Each Instructions contributor emits at most one XML-delimited block: a non-empty block carries its named tag; an empty contributor (e.g. the catalog with no skills discovered) contributes nothing and the composer skips it. The scaffold-frame block is registered first. The full registration order is fixed so the composed string is byte-stable but, beyond frame-first, not semantically load-bearing. A reorder is caught by a contributor-order assertion in the scaffold test suite, not by the anti-revert content-hash test: G8's hash covers only the instructions_frame_text literal, which a reorder leaves byte-identical. Content anti-revert and structural anti-revert are separate guarantees with separate tests.

Journeys

J-skills-01

Author and activate a skill

Pre-conditions

  • A project skill directory exists with one valid SKILL.md.

Steps

  1. Start an agent; the skills catalog appears in Run.instructions.
  2. The catalog lists the authored skill's name and description.
  3. The agent calls Skill(name) for that skill.
  4. The tool result contains the frontmatter-stripped body inside the structural wrapper.
  5. The activation persists as an ordinary tool-call/tool-return pair in the Run's messages.
J-skills-02

Untrusted tool result is wrapped

Pre-conditions

  • A test-only tool registered with trust_class="untrusted".

Steps

  1. The agent invokes the untrusted tool.
  2. The tool result the model receives is enclosed in <untrusted_content>…</untrusted_content>.
  3. A trusted tool's result in the same run carries no such markers.

Resolved questions

#QuestionDecision
Q1Build the <untrusted_content> wrapping path now, or defer until a real untrusted tool exists?Build now; exercise it with a test-only untrusted tool fixture: the seam is complete and proven the day it lands, and ToolCapability is the single convergence point per arch.
Q2Is the Instructions-composer migration its own prerequisite spec or part of this one?Part of this spec: the skills catalog must be a composer contributor, and capture invariant I2 requires the composer to own all Instructions content; while instruction_text rides the static slot it bypasses capture.
Q3What does Skill(name) return on activation?Frontmatter-stripped body, structurally wrapped, as a tool_result: follows the Agent Skills standard's dedicated-tool activation pattern.
Q4Where does the persona block sit relative to the skills catalog?Position made non-load-bearing. Every block is XML-tagged, so the model finds a section by its tag. Only frame-first is fixed (it carries the interpretive conventions). The scaffold is modeled on Letta, which builds the context window from memory blocks addressed by a unique label rather than by position; the XML-tagged Instructions blocks are the same idea.
Q5Is the scaffold frame fixed in code or config-overridable?Config-overridable: instructions_frame_text carries a scaffold default a deployment may replace; its anti-revert hash is pinned for the default.
Q6Should the Skill tool be registered when no skills are discovered?No: an enum with zero members is invalid and a no-op tool is noise; the tool is registered only when ≥1 skill exists.
Q7One reusable contributor function, or a distinct function per block?A shared wrap(tag, text) primitive plus a static_block helper for literal-text blocks (frame, persona); the skills catalog is the one bespoke contributor because its text is computed.

Testing strategy

  • Unit-tested: ToolCapability: tools registered with each trust_class/side_effect_class value; assert pass-through, gate routing, and <untrusted_content> wrapping per branch.
  • Unit-tested: metadata fail-fast: register a tool missing a metadata field; assert agent construction raises.
  • Unit-tested: SkillRegistry discovery and parsing: fixture skill tree with a valid skill, a project/user name collision, and a malformed SKILL.md; assert the registry resolves the collision project-wins and skips the malformed file with a warning.
  • Unit-tested: Instructions composer: assert the frame block is first, each contributor's output is an XML-delimited block carrying its named tag, and Run.instructions equals the composed string (invariant I2).
  • Journey-tested: skill authoring → catalog → activation: J-skills-01 runs against a live agent over a fixture skill tree.
  • Journey-tested: untrusted-content wrapping: J-skills-02 runs with a test-only untrusted tool fixture.
  • Anti-revert: the default instructions_frame_text: pinned content-hash test, updated in the same commit as any change to the literal.
  • Not tested: the L1 side-effect gate's approval logic: it is a deliberate no-op at L1; only the routing into it is tested.

Forward-compatibility

  • The persona contributor reads config.persona_text today and is expected to swap to MemoryStore.core_blocks() when MemoryStore lands. The contributor boundary: a function of deps returning a string: is the stable interface.
  • The memory-index contributor registers after the frame, alongside persona and the skills catalog. Because blocks are XML-tagged, its exact position is not load-bearing: invariant I4 constrains only "after the frame."
  • The L1 side-effect gate is a no-op that L3 replaces with a human approval queue. Callers route through the gate today so that swap is internal to ToolCapability.
  • The tool-metadata convention is the forward contract for every future tool (memory tools, web fetch, retrieval). New untrusted tools need no ToolCapability change: they declare trust_class="untrusted" and the existing wrapping path handles them.
