Serving a pydantic-ai Agent over HTTP

An Agent is just a function: prompt in, text (or structured object) out. Wrapping it in an HTTP endpoint sounds trivial — and the one-line version is. But there are six or seven distinct shapes that endpoint can take, and the choice you make at this layer leaks into your client code, your observability, your test seams, and how comfortable your endpoint is to retrofit later.

First answer first: if your framework ships a wire format (pydantic-ai ships AG-UI), use it. That's approach 0 below — the one this repo currently runs. The remaining eight approaches are the hand-rolled options you'd be choosing between if the framework didn't have an opinion. Knowing them is still worthwhile: it's what makes "use the built-in" a deliberate decision instead of a default-shrug, and it's the menu you'd reach for if you ever needed to drop the adapter (custom request validation, per-request auth, a wire shape AG-UI doesn't cover).

Approach 00

AG-UI native current

Delegate wire-format concerns to the framework. Server speaks AG-UI via pydantic_ai.ui.ag_ui.AGUIAdapter; a tiny Python adapter (assistant.client.AssistantClient) restores the one-liner UX for non-browser callers.

from fastapi import Depends, FastAPI, Request
from fastapi.responses import Response
from pydantic_ai import Agent
from pydantic_ai.ui.ag_ui import AGUIAdapter

@app.post("/chat")
async def chat(
    request: Request,
    agent: Agent[None, str] = Depends(get_agent),
) -> Response:
    return await AGUIAdapter.dispatch_request(request, agent=agent)

Wire shape

HTTP/1.1 200 OK
Content-Type: text/event-stream

data: {"type":"RUN_STARTED","threadId":"...","runId":"...","timestamp":...}

data: {"type":"TEXT_MESSAGE_START","messageId":"...","role":"assistant","timestamp":...}

data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"...","delta":"po","timestamp":...}

data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"...","delta":"ng","timestamp":...}

data: {"type":"TEXT_MESSAGE_END","messageId":"...","timestamp":...}

data: {"type":"RUN_FINISHED","threadId":"...","runId":"...","timestamp":...}

The request body is a RunAgentInput: thread_id, run_id, messages, plus required-empty tools, context, and nullable state / forwarded_props. Full field list and event-type strings live in docs/references/ag-ui-surface.md.

When to reach for it

  • Your framework ships a wire format. Using the vendored one is the right default — you stop owning SSE framing, event encoding, and the version compatibility surface.
  • You want a browser path (CopilotKit / AG-UI React components) and a Python path on the same endpoint without dual-endpoint plumbing.
  • Structured event vocabulary matters: TEXT_MESSAGE_CONTENT vs TOOL_CALL_START vs STATE_SNAPSHOT all flow on one stream with discriminated types.

When to avoid

  • You need custom request validation, per-thread quotas, or auth that must fire before the agent runs. dispatch_request is convenience-shaped; drop to the manual build_run_input / run_stream / encode_stream flow when you outgrow it.
  • Your wire is constrained to something AG-UI doesn't cover (custom binary protocol, fixed-shape JSON for a legacy consumer).

Pros

  • Zero hand-rolled SSE framing. Wire format drift is the framework's problem.
  • One endpoint, many consumers (browser, Python, future TUI).
  • Test seam unchanged: Depends(get_agent) + app.dependency_overrides still inject TestModel-backed Agents (verified in tests/test_smoke.py).
  • Richer event vocabulary available when you need it (tools, thinking, state, reasoning — see ag_ui.core.EventType).

Cons

  • RunAgentInput is verbose to type. The assistant.client.AssistantClient adapter exists exactly to hide that.
  • Wire-format coupling to pydantic-ai's minor versions. Re-verify ag-ui-surface.md at every bump.
  • You don't own the SSE framing. If something breaks at that layer, the failure is one library deep.
Approach 01

Non-streaming JSON

Wait for the whole response, return one JSON body.

@app.post("/chat")
async def chat(req: ChatRequest, agent: Agent[None, str] = Depends(get_agent)):
    result = await agent.run(req.message)
    return {
        "output": result.output,
        "usage": result.usage().model_dump(),
        "messages": [m.model_dump() for m in result.all_messages()],
    }

Wire shape

HTTP/1.1 200 OK
Content-Type: application/json

{"output": "pong", "usage": {...}, "messages": [...]}

When to reach for it

  • Server-to-server APIs where the consumer parses JSON, not SSE.
  • Short responses (a paragraph or less) where streaming buys you nothing.
  • Anywhere the client is going to wait for the full text anyway before doing anything with it (RAG pipelines, scoring jobs, classifiers).
  • When you need the full result object — usage, messages, tool calls — and don't want to reassemble it on the client.

When to avoid

  • Interactive UIs. Users hate watching a spinner for 8 seconds when they could be reading the first paragraph.
  • Long outputs (anything > ~2 seconds of generation). Proxy timeouts and perceived-latency both bite.
  • When you'd like the option to cancel mid-generation. Plain JSON gives the server no signal to stop the model call.

Pros

  • Simplest possible client (one fetch, one .json()).
  • Trivially cacheable, retryable, idempotent-with-a-key.
  • Full result available — usage, messages, structured output, tool trace.
  • Works through any HTTP infra; no SSE-aware proxy required.

Cons

  • Time-to-first-byte = time-to-full-response. Bad UX for chat.
  • Big payloads stall behind buffer flushes at every proxy hop.
  • No way for the client to react to partial output (early-stop, render-as-you-go).
Approach 02

SSE with delta tokens launch-time baseline

Stream new text as it arrives. Each event carries only the new piece. This is what /chat spoke before the AG-UI migration; the commit immediately prior to the migration is tagged hand-rolled-sse on init/scaffold if you ever want to read that code as a reference.

@app.post("/chat")
async def chat(req: ChatRequest, agent: Agent[None, str] = Depends(get_agent)):
    async def stream():
        async with agent.run_stream(req.message) as result:
            async for delta in result.stream_text(delta=True):
                yield f"event: token\ndata: {json.dumps({'delta': delta})}\n\n".encode()
        yield b"event: done\ndata: done\n\n"
    return StreamingResponse(stream(), media_type="text/event-stream")

Wire shape

HTTP/1.1 200 OK
Content-Type: text/event-stream

event: token
data: {"delta": "po"}

event: token
data: {"delta": "ng"}

event: done
data: done

When to reach for it

  • Interactive chat UIs in a browser. EventSource is built-in.
  • You want low time-to-first-token and don't care about per-event metadata.
  • Your client genuinely streams to the user as deltas arrive (typing effect).

When to avoid

  • Non-browser clients where SSE libraries are flaky or absent (some CLI tools, mobile networking stacks).
  • Clients that just want the final text — they'll have to reassemble for no reason.
  • Behind aggressive corporate proxies that buffer responses and break the streaming illusion.

Pros

  • Time-to-first-token is whatever the model gives you — typically < 500ms.
  • Built-in browser support via EventSource (auto-reconnect, named events).
  • Cheap to implement and debug: curl -N shows it in plain text.

Cons

  • One-way only. To cancel, the client has to drop the connection.
  • The final assembled output and usage metadata are discarded unless you bolt them on — see approach 4.
  • EventSource in the browser is GET-only; POST-with-SSE needs fetch + a manual stream reader.
Approach 03

SSE with cumulative text

Stream the full-text-so-far on every chunk, not just the delta.

async def stream():
    async with agent.run_stream(req.message) as result:
        async for text in result.stream_text(delta=False):   # <-- delta=False
            yield f"event: text\ndata: {json.dumps({'text': text})}\n\n".encode()
    yield b"event: done\ndata: done\n\n"

Each text here is the complete response so far: "po", then "pong", then "pong!". The client just renders the latest one, no concatenation needed.

When to reach for it

  • Rendering markdown live. Markdown parsers don't tolerate partial input mid-token (think a half-open code fence); re-parsing the cumulative text on each chunk is simpler and more robust than maintaining a stateful incremental parser.
  • Clients that can lose chunks (mobile networks, retries mid-stream). The last chunk you receive is always the truth.

When to avoid

  • Long outputs. Bandwidth grows as O(n²) — each chunk re-sends everything before it.
  • Clients doing a typing-effect animation. They need deltas, not snapshots.
Approach 04

SSE deltas + final result event recommended next step

Stream tokens for UX; emit one closing event with the assembled output and metadata.

async def stream():
    async with agent.run_stream(req.message) as result:
        async for delta in result.stream_text(delta=True):
            yield f"event: token\ndata: {json.dumps({'delta': delta})}\n\n".encode()
        # After the loop exhausts, result still holds the full state.
        final = {
            "output": result.get_output(),
            "usage": result.usage().model_dump(),
        }
        yield f"event: result\ndata: {json.dumps(final)}\n\n".encode()
    yield b"event: done\ndata: done\n\n"

Same wire as approach 2 with one extra event before done:

event: token
data: {"delta": "po"}

event: token
data: {"delta": "ng"}

event: result
data: {"output": "pong", "usage": {"requests": 1, "request_tokens": 12, "response_tokens": 1, "total_tokens": 13}}

event: done
data: done
Why this is the upgrade

Streaming clients can ignore the result event and keep their delta-rendering logic. Storage / replay / observability clients can listen for just the result event and skip the deltas. You add one event, you make the endpoint useful for two consumer shapes, and you stop throwing away the metadata pydantic-ai already computed.

Tradeoff to know about

  • The wire format gains a new event type. Anything (a test, a client, a contract) that asserts the exact stream of events needs updating.
  • The closing burst (result immediately followed by done) is two events the client sees back-to-back. If a client treats done as "now I can read the final result from local state," moving the result event before done just works.
Approach 05

NDJSON over chunked encoding

One JSON object per line. Like SSE, but cheaper to parse and friendlier to non-browser clients.

async def stream():
    async with agent.run_stream(req.message) as result:
        async for delta in result.stream_text(delta=True):
            yield (json.dumps({"type": "token", "delta": delta}) + "\n").encode()
        yield (json.dumps({"type": "result", "output": result.get_output()}) + "\n").encode()
return StreamingResponse(stream(), media_type="application/x-ndjson")

Wire shape

HTTP/1.1 200 OK
Content-Type: application/x-ndjson
Transfer-Encoding: chunked

{"type": "token", "delta": "po"}
{"type": "token", "delta": "ng"}
{"type": "result", "output": "pong"}

When to reach for it

  • CLI clients, scripts, and any consumer that already does line-buffered IO. curl ... | jq -c works out of the box.
  • Backends consuming the stream where the SSE protocol is overkill (you don't need event names, just typed payloads).
  • Apple platforms, where EventSource support has historically been spotty and JSON-lines is universal.

When to avoid

  • Browser clients. There's no native NDJSON reader; you'd write the same fetch-and-parse loop you'd write for SSE anyway, but without the named-event ergonomics.
  • Anywhere you want client-side automatic reconnect — that's an SSE feature, not an NDJSON one.
Approach 06

WebSocket

A bidirectional, long-lived connection. The client can send more messages or cancel mid-generation.

@app.websocket("/chat")
async def chat_ws(ws: WebSocket, agent: Agent[None, str] = Depends(get_agent)):
    await ws.accept()
    try:
        while True:
            data = await ws.receive_json()    # {"type": "user", "message": "..."}
            if data["type"] == "cancel":
                continue                       # handled by the outer loop wiring
            async with agent.run_stream(data["message"]) as result:
                async for delta in result.stream_text(delta=True):
                    await ws.send_json({"type": "token", "delta": delta})
                await ws.send_json({"type": "result", "output": result.get_output()})
    except WebSocketDisconnect:
        return

When to reach for it

  • Multi-turn chat where the connection persists across turns (avoiding TCP/TLS handshakes per message).
  • You need genuine bidirectionality: cancel buttons, mid-stream user interjections, server-pushed events between user turns.
  • Server-pushed background events: tool-call traces, "agent is thinking" indicators, partial structured-output validations.

When to avoid

  • The endpoint is genuinely one-shot. WebSocket adds a handshake, a framing layer, and a state machine for no benefit.
  • You want HTTP-shaped observability (status codes, response times per request). WS connections don't slot into standard request-metric dashboards.
  • Your infrastructure doesn't love long-lived connections — serverless platforms, CDN proxies, browser tabs that go to background.
Watch out

WS dependency injection in FastAPI works but is more limited than HTTP routes — app.dependency_overrides still works for the test seam, but middleware, auth helpers, and rate-limiters often need WS-specific equivalents.

Approach 07

Streaming structured output

When output_type is a pydantic model, stream partial validations as fields fill in.

class TicketDraft(BaseModel):
    title: str
    severity: Literal["low", "med", "high"]
    body: str

agent = Agent(model=..., output_type=TicketDraft)

async def stream():
    async with agent.run_stream(req.message) as result:
        async for partial in result.stream():       # yields partial TicketDraft objects
            yield f"event: partial\ndata: {partial.model_dump_json()}\n\n".encode()
        final = result.get_output()
        yield f"event: result\ndata: {final.model_dump_json()}\n\n".encode()
    yield b"event: done\ndata: done\n\n"

The client sees a sequence of progressively-more-complete TicketDraft objects: first just a title, then title + severity, then all three fields. UIs that render structured forms can light up each field as it lands.

When to reach for it

  • Structured output (a pydantic model with multiple fields).
  • UIs that show a live-updating form, table row, or card.
  • Pipelines that can act on partial-but-valid data (e.g., start a database lookup as soon as the id field arrives).

When to avoid

  • Plain text output. There's nothing structured to stream.
  • Strict consumers that can't tolerate intermediate states. pydantic's partial validation is permissive (missing required fields, narrow types as the parse progresses); your consumer code has to be too.
Approach 08

Async job + poll

POST creates a job and returns immediately. The client polls (or subscribes) for completion.

@app.post("/chat")
async def chat(req: ChatRequest, bg: BackgroundTasks) -> dict:
    run_id = uuid4().hex
    bg.add_task(run_agent_and_persist, run_id, req.message)
    return {"run_id": run_id, "status_url": f"/chat/{run_id}"}

@app.get("/chat/{run_id}")
async def chat_status(run_id: str) -> dict:
    return read_run(run_id)   # {"status": "pending"|"done", "output": ...}

When to reach for it

  • Runs that may take more than a few seconds and clients that can poll comfortably (cron jobs, background workers).
  • You want the result to survive the client disconnecting (close a laptop, kick off agent, come back tomorrow).
  • You're building agent runs that branch into tools, sub-agents, or sleeps; you want the run as a first-class entity with its own lifecycle.

When to avoid

  • Interactive chat. Polling is a terrible UX.
  • You don't have a persistence layer yet. Async jobs require storage; without it the "job" lives in process memory and dies with the worker.
Bridges

This pattern composes nicely with the others: the GET endpoint can return JSON (approach 1), or hand back an SSE stream (approach 2) that replays the deltas captured to storage. The job model decouples "when the work happens" from "how the client consumes it."

"Browser-native" means the browser exposes a built-in API for the protocol (fetch for JSON, EventSource for SSE, WebSocket for WS). "Partial" SSE rows reflect that EventSource is GET-only — POST-with-SSE requires a manual fetch-and-parse loop.

The "final result" question

The current /chat implementation streams text deltas and then closes with event: done. After the async for exhausts, pydantic-ai's result object still holds everything the model returned:

  • result.get_output() — the full assembled output (or the validated structured object, if output_type is a pydantic model)
  • result.usage() — token counts, request counts, cost-relevant metadata
  • result.all_messages() — the full conversation including system/user/assistant turns and any tool calls
  • result.new_messages() — just the assistant turn produced by this run (handy for appending to a stored transcript)

We throw all of that away. For a phase-0 endpoint with no persistence, that's defensible — but as soon as we want to log a run for evals, replay a transcript, or show the user a "this answer cost N tokens" indicator, the metadata becomes load-bearing.

Recommendation

Adopt approach 4 (SSE deltas + final event: result) when persistence or observability lands. It's a one-event addition to the wire, doesn't break existing delta consumers, and lets a future logging or eval pipeline subscribe to just the result event. Combined with the planned vault transcript storage, this is the cleanest seam to "every run is observable" without paying for it in phase 0.

The alternative — keeping deltas-only on /chat and exposing the final result via a separate GET /runs/{run_id} — is the right shape if and only if you're already on approach 8. For a synchronous streaming endpoint, splitting it across two HTTP requests is more moving parts than it's worth.

Decision guide

If you're picking once and moving on:

  • Browser chat UI? Approach 2 today, approach 4 the moment you want usage/metadata.
  • Server-to-server, short responses? Approach 1. Don't pay streaming complexity for nothing.
  • CLI client? Approach 5 (NDJSON). One while read line away from working.
  • Multi-turn, with cancel? Approach 6 (WebSocket).
  • Long-running, retryable, survives client drop? Approach 8, with one of the others bolted on for the GET side.
  • Structured output that fills in over time? Approach 7.

The honest truth: most teams default to approach 2 because every chat-UI tutorial uses it, hit metadata pain at month three, then either bolt on approach 4 (small win, no breaking change) or split into approach 8 + a polling client (bigger refactor, but unlocks persistence and replay). Skipping the "discover the metadata problem the hard way" step is the main thing this document is for.
