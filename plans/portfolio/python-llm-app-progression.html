Portfolio · vision & architecture

Understanding how Python LLM apps grow into agents

A 10-level public GitHub progression that builds one growing personal LLM assistant, hand-rolled on OpenAI and FastAPI. Conversation is the substrate. The Obsidian vault is the memory store. Tools are added when the assistant earns them.

Intent

Goal

Build a public, ten-project portfolio that demonstrates a coherent progression from a daily-use chat assistant to a self-improving agent. Each level teaches new concepts, every project is useful to a human (starting with the author), and every project builds on the prior one. The throughline is one growing personal assistant, not ten unrelated apps.

Non-goals
  • Enterprise-scale production: “production” here means publishable, polished, and useful, not SLA-grade.
  • Recreating the analog repos we studied: we learn from them and write our own.
  • Multi-provider portability: OpenAI only for dev to keep costs and noise down.
  • Framework adoption: no LangChain, CrewAI, or OpenAI Agents SDK. Hand-rolled on the raw OpenAI SDK is the point.
Key insight

The hardest constraint is no SME for evals. That makes Eval strategy the spine of every level, not an afterthought. L5 is the explicit pivot. We build the eval muscle on simple apps before agents make it hard.

Goals

  1. G1
    Each level introduces exactly one new technical concept, feature, or agent paradigm the author can articulate and defend in an interview.
  2. G2
    Each level produces a working app a human would actually use: not a notebook, not a demo.
  3. G3
    Each level publishes as a standalone GitHub repo with locked deps, README, CI smoke test, and a logged-LLM-calls store (see I5: every call, tool invocation, and cost traced to a local SQLite or JSONL store for annotation, failure analysis, and evals).
  4. G4
    Every level uses the same OpenAI + FastAPI + Python stack and a hand-rolled agent harness.
  5. G5
    Every level contributes to one growing personal assistant. The Obsidian vault is the memory substrate; tools are added when they earn their place; no level is a standalone data pipeline.
  6. G6
    Each level is annotate-able, supports failure analysis, and is evaluable without a domain expert (the workflow Hamel Husain documents at hamel.dev): tasks have verifiable ground truth, judged-by-rubric outputs, or trajectory evals.

Non-goals

  1. NG1
    Multi-model / multi-provider production support. OpenAI is the only build-phase provider. We do keep a thin provider-abstraction seam (see ADR-006) so future expansion is cheap, but we do not implement a second provider in this curriculum. litellm may enter at L5 if eval comparisons need cross-provider judges.
  2. NG2
    Framework cargo-culting: LangChain, CrewAI, LlamaIndex, OpenAI Agents SDK. We learn from them, we don’t use them. (Pydantic-AI and Letta are not in this list; see I7 for why we reserve the right to revert.)
  3. NG3
    A unified frontend across all levels. CLI / TUI / web is per-level; the Obsidian plugin is the north star, not a deliverable per level.
  4. NG4
    Polishing already-shipped levels when a new level lands. Each repo is “done at the time of its commit”: we add to the portfolio, we don’t maintain ten things.

Invariants

  1. I1
    Every project has an LLM loop with more than one call: even L1 retries on validation failure.
  2. I2
    Every project has some form of memory. The kind varies (in-context, always-loaded AGENTS.md-style, semantic, agent-owned, episodic, procedural) and is the central learning artifact of that level.
  3. I3
    Every project has a FastAPI backend. The frontend is whatever fits (CLI, TUI, or web). The API stays consistent so any future surface (Obsidian plugin, mobile app, hosted web UI) plugs into the same backend.
  4. I4
    Every project ships with an eval suite proportional to its complexity. L1 needs ‘X’ labeled examples; L10 needs a closed-loop self-evaluator.
  5. I5
    Trace. All LLM calls, tool invocations, and costs are logged to a local SQLite or JSONL store. This is the source for annotations, failure analysis, and evals. Non-negotiable from L1.
  6. I6
    Every markdown file our agents write to the vault that contains agent-readable instructions is Skills-spec compliant. (Open question: verify the Skills-spec frontmatter is also Obsidian-compatible so the same file renders cleanly in the vault.)
  7. I7
    Goal is no agent framework imports outside openai. Tool schemas, the loop, retries, and memory are written by us, in our repo. (We may revert to Pydantic-AI or Letta at some point, and we are likely to take detours to explore how the frameworks would make our lives easier. The default stance is hand-rolled; the detours are deliberate exceptions, not drift.)
  8. I8
    Writes to external systems (Bluesky posts/replies/likes/follows, future tool integrations) are scoped to bot or sandbox accounts owned by the author. Reads can be global, subject to platform terms and rate limits. Larger public-figure accounts that opt into bot interaction can be allowlisted; default-deny on writes otherwise. Protects against accidental notification spam to non-consenting users and keeps the portfolio defensible.
  9. I9
    App code never imports a provider SDK directly. Chat, streaming, and structured-output calls route through pydantic-ai’s Agent / Model interface (per ADR-006). Embeddings route through the indexer’s provider-agnostic interface (e.g., chromadb’s embedding functions). Provider swap is a model-string change in config, not a code change.
  10. I10
    Every API call identifies its calling principal (human user or agent). Conversations, semantic-recall corpora, user-fact memories, and traces are partitioned per principal; skills and shared project knowledge are global. Multi-principal from L1: designed in, not retrofitted. See ADR-007.

Context

The 2026 agent landscape is crowded with frameworks but thin on production references. The production references that do exist mostly wrote their own harness.

The most-studied agent frameworks in Python (LangChain/LangGraph, CrewAI, AutoGen/AG2, smolagents, Agno, DSPy, Strands, Atomic Agents, Haystack, Letta, Pydantic-AI, OpenAI Agents SDK, Claude Agent SDK) share one structural property: their public production rosters thin out fast under verification. LangChain/LangGraph has the deepest (Klarna, Replit, LinkedIn, Elastic). Pydantic-AI has a handful (MindsDB, Sophos, Boosted.ai) but mostly from the vendor’s own case-study page. Letta has Bilt and a few others. The two new agent SDKs (OpenAI Agents, Claude Agent) have no verified third-party shipped products yet. Both are under fourteen months old as public libraries.

Meanwhile, the most respected shipped agentic products (Claude Code, Cursor, Cognition’s Devin, Cline, Continue, Aider, OpenHands) are built on the raw provider SDKs with hand-rolled harnesses. Anthropic, Cursor, and Cognition have all published essays explaining the rationale: control over context, debuggability, model-specific tool formats, and (Anthropic’s point) the desire to delete code when new models land.

Why this matters for the portfolio

If the most credible shipped products avoid frameworks, the hand-rolled approach is itself the marketable skill. The portfolio’s narrative is not “I can use Pydantic-AI”: it’s “I built a small Claude Code on my own laptop and I understand every line.”

Assumptions
  • OpenAI’s pricing on small and medium models stays cheap enough that an author can run hundreds of eval runs per level without budget pressure.
  • Markdown with YAML frontmatter and wikilink-style cross-references remains a viable substrate for agent-readable memory through the portfolio’s build window. The vault is just a folder of files; if Obsidian itself shifts, the underlying format is portable.
  • The agentskills.io spec stays the cross-vendor convergence point for agent skill files.

Design

Three macro-phases. The personal-assistant substrate, the eval pivot, then tools and agents.

Phases

Macro-phase 1

Assistant substrate (L1–L4)

  • Chat, semantic recall, declarative memory, procedural skills
  • Conversation-first; vault as the memory store
  • Each level is useful daily before the next ships
Macro-phase 2

Eval pivot (L5)

  • Small hand-built eval harness
  • Retroactively evaluates L1–L4
  • Substrate for L6–L10
Macro-phase 3

Tools and agents (L6–L10)

  • Hand-rolled ReAct on the raw SDK
  • Tools added when the assistant earns them
  • Agent-written skills, long-horizon work, self-improvement

Levels

One subsection per level. Each names the new concept, the primary analog we studied, the imports we’ll write against, and the eval approach. L1–L4 build the assistant’s conversational substrate; L5 is the eval pivot; L6–L10 add tools and agentic behaviors on top.

L1 · Chat with persistent transcript

Concept. Streaming responses, multi-turn state, SQLite-backed session history. Each session is also written back to the vault as a session note. Multi-call (I1) is satisfied because the assistant retrieves a small slice of prior-session context (recency + keyword) before answering. Every session is tagged with a principal (I10): the local CLI defaults to the OS user; HTTP clients (Claude Code, future agents) pass an API key that resolves to their principal ID.

Use case. A CLI chat you use daily. Conversations are searchable, persist across sessions, and accumulate as the assistant’s long-term ground truth about how each principal (you, Claude Code) talks to it.

Primary analog. Azure-Samples/rag-postgres-openai-python (490 stars) for FastAPI + streaming + persistence. Secondary: basicmachines-co/basic-memory (3k stars) for industrial-grade conversation-plus-notes persistence; simonw/llm (12k stars) for the CLI shape.

Imports. pydantic_ai.Agent (per ADR-006), fastapi.responses.StreamingResponse, sqlmodel, uuid, pydantic-settings, typer.

Eval. Stream-cancel correctness (deterministic). Persistence round-trip (deterministic). Latency and tokens-per-second baseline. Multi-turn coherence (judge).

L2 · Semantic recall over your conversations

Concept. Embeddings, chunking strategy, hybrid retrieval (vector + BM25), applied to your accumulated chat history. Before generating, the assistant retrieves the most relevant past exchanges and injects them as context. The corpus is you talking to it, not an external dataset.

Use case. “We talked about X last week, where did we land?” answered without scrolling. The assistant brings forward the relevant past on its own.

Primary analog. benmaster82/Kwipu (Python, 80 stars) for the hybrid-retrieval shape. basic-memory for the conversation-corpus shape. Vasallo94/ObsidianRAG (79 stars) for the indexer-reads-the-vault pattern.

Imports. chromadb (or sqlite-vec), rank_bm25.BM25Okapi, tiktoken, watchdog (re-index on vault change).

Eval. Retrieval metrics (recall@k, MRR) on bootstrapped (query → expected past-exchange) pairs sampled from your own chat history. Faithfulness when the retrieved context is summarized into the response.

L3 · Declarative memory

Concept. Distinct from session history: facts about each principal (preferences, ongoing projects, opinions, recurring entities) distilled and stored as always-loaded markdown. Memory is principal-scoped per I10: facts about Scott live in his memory folder, facts about Claude Code (its quirks, prior delegations, observed failure modes) live in its own. Shared project knowledge sits in a global namespace. Read on session start. Updated by the assistant when new facts emerge or old ones contradict.

Use case. The assistant knows you between sessions (“Scott is working on the portfolio plan, defaults to uv, dislikes em-dashes”) without you re-saying it every time. When Claude Code calls the assistant, it can recall what it knows about Claude Code as a separate principal.

Primary analog. cline/cline (61k stars) Memory Bank pattern (.clinerules/ as always-loaded procedural memory). Anthropic’s own AGENTS.md / CLAUDE.md convention as the canonical shape. Ar9av/obsidian-wiki (1.4k stars) for vault conventions (manifest, hot.md, staging, provenance markers).

Imports. yaml, frontmatter, the vault-write primitives from L1.

Eval. Multi-session probe: facts planted in session N must surface in session N+1. Memory-decay correctness: stale facts dropped when contradicted, with a citation to the contradicting exchange.

L4 · Skills

Concept. Procedural memory. SKILL.md files in the vault that describe how to do recurring tasks. Loaded on match per the Skills-spec progressive disclosure model. The assistant has opinions about how to do common things, and reaches for those opinions when the request matches.

Use case. You ask for a PR review or a project scaffold; the assistant pulls the matching skill, follows its playbook, and asks the right clarifying questions instead of re-deriving the process every time.

Primary analog. kepano/obsidian-skills (32k stars): Obsidian CEO’s official skill bundle, our format target. Anthropic Agent Skills as the canonical schema. cline/cline Memory Bank for the procedural-memory shape.

Imports. yaml, the SKILL.md loader (hand-written), watchdog.

Eval. Skill-match precision (did the right skill load for the query?). Skill-following adherence (did the response follow the skill’s steps?).

L5 · Eval harness : the pivot

Concept. Golden datasets, LLM-as-judge with rubric, regression detection. Per-intent bucketing (structural stratification applied to chat intents and tool calls). L5 retroactively evaluates L1–L4 and becomes the trace-reading substrate every later level depends on.

Use case. A CLI to compare prompts, models, and skill versions across the prior four levels’ eval suites.

Primary analog. meshkovQA/Eval-ai-library (Python, 44 stars) for size and shape. Read openai/evals (18k stars) for the YAML + grader pattern but don’t template: too big. ragas (14k stars) for chat-specific metrics applied to L1–L2.

Imports. yaml, jsonlines, rich.console.Console, scipy.stats.spearmanr (judge-vs-truth correlation), pandas, optionally litellm for cross-provider judge comparisons.

Eval. This level is the eval. Validation: judge correlation with held-out human spot checks on a 10-example slice.

L6 · First tool, hand-rolled ReAct

Concept. The agent loop itself, on the raw OpenAI SDK. Tool schemas generated from Python function signatures, JSON parsing, looping until done. One tool first, kept tight. The first level where the assistant reaches outside the conversation.

Use case. The assistant gains the ability to fetch a URL (or run a web search). The conversation surface starts demanding outside reach, and the assistant earns its first tool.

Primary analog. simonw/llm’s _tools.py as the battle-tested reference. gptme/gptme (4.3k stars) for the loop shape: a tight, readable Python function.

Imports. httpx, tavily-python or duckduckgo-search, inspect (introspect function signatures into tool schemas).

Eval. Trajectory (right tool, right args, right number of iterations). Task success on a small fetch-or-search dataset.

L7 · Multi-tool agent

Concept. Several tools (web search, web fetch, vault search, time/date). Agent decides which to call and in what order. The memory and skills from L3 and L4 govern when to reach out: if the assistant already knows the answer, no tool call should fire.

Use case. “Plan my afternoon based on what’s in my calendar, what’s on Hacker News, and what we said yesterday.” Multi-tool orchestration with memory-aware suppression of unnecessary calls.

Primary analog. gptme/gptme (multi-tool loop). NirDiamant/Agent_Memory_Techniques (378 stars) for memory-aware tool routing patterns.

Imports. Continuation of L6 tools plus a router/dispatch layer; sqlite3 for tool-call cache.

Eval. Trajectory at higher cardinality (did it skip tools it didn’t need?). Task success on multi-tool tasks. Token efficiency: did memory and skills suppress unnecessary tool calls?

L8 · Agent-written skills

Concept. The assistant observes patterns across conversations and proposes new SKILL.md drafts. The user approves; the drafts enter the catalog. Procedural learning that closes the gap between L4 (hand-written skills) and L10 (self-improvement).

Use case. After the third time you ask for the same kind of PR review, the assistant says “I noticed a pattern; here’s a draft skill for it; approve?”

Primary analog. cline/cline Memory Bank (.clinerules/ written by the agent itself). breferrari/obsidian-mind (2.6k stars) for vault templates. kepano/obsidian-skills as the format target.

Framing correction

Ar9av/obsidian-wiki and kepano/obsidian-skills are not Python agent frameworks: they’re markdown-only skill collections that run on Claude Code/Cursor/Codex. We borrow their conventions; we don’t port them. L8’s skill-drafting loop is original code.

Imports. yaml, watchdog, the L4 skill loader. The agent reads its own L5 traces to detect patterns.

Eval. Skill quality: do agent-drafted skills pass the L4 skill-following eval? User acceptance rate: do you actually approve them?

L9 · Planner + worker, long-horizon

Concept. Two-role agent split: planner (read-only, proposes) and worker (applies edits). Context compaction. Hand-off contracts. The first level where the assistant takes on tasks that span hours and many steps. Bluesky enters here as a tool the agent calls, subject to I8 (writes are bot-scoped).

Use case. “Build me a Bluesky dossier on @karpathy.” Or: “Audit my vault for orphan notes and propose links across hundreds of files in one pass.” Long-horizon work the assistant can finish unattended and hand back a result you review.

Primary analog. sst/opencode (161k stars): cleanest open plan/build split. Secondary: SWE-agent (19k stars) mini variant (~100 LOC) for the academic-clean planner→tool loop.

Imports. atproto (Bluesky thread/profile reads; bot-scoped writes), networkx (vault graph analysis), asyncio, collections.deque (work queue).

Eval. Task success on held-out long-horizon tasks (planted Bluesky scenarios using bot fixtures; planted vault setups with known orphans). Trajectory: did the planner stay read-only?

L10 · Self-improving agent

Concept. Agent reads its own L5 traces, identifies failure clusters per skill, updates its L4 and L8 skills accordingly. Capstone: closes the eval/improvement loop.

Use case. The assistant gets measurably better at the tasks you actually ask for, without manual prompt-tuning.

Primary analog. NousResearch/hermes-agent-self-evolution (3.4k stars): the only repo whose architecture matches the spec literally. Engine: stanfordnlp/dspy (34k stars) GEPA optimizer over the skill markdown files. Fallback: EvoAgentX/EvoAgentX (3k stars).

Imports. dspy, jsonlines (trace store), sqlite3 (trace index). Reuses L5 eval harness and L4/L8 skill files.

Eval. Meta-eval: does the agent’s self-improvement actually improve held-out task performance? Spearman ρ on a held-out set.

Known wrinkle

L10 has no clean production analog. Plan it as a capstone writeup with a thinner code surface than L8 or L9. ~50% blog post, 50% code. Hermes itself last committed 2026-03-29: stale relative to the rest of the analog roster.

Decisions

ADR-001

Use OpenAI only for the build phase

Context. OpenAI, Anthropic, Gemini, and open-weight models all viable.

Decision. OpenAI only for L1–L10. litellm may enter at L5 to enable cross-provider evals: never earlier.

Consequences. Cheaper iteration on small models; fewer provider-quirk distractions; portfolio doesn’t demonstrate provider abstraction (acceptable: see NG1).

Alternatives
Anthropic only; litellm from L1; local model via Ollama
Owner
Scott
ADR-002

Hand-roll the agent harness on the raw SDK

Context. The agent-framework landscape is crowded; the most credible shipped products avoid frameworks.

Decision. Write our own loop, tool schemas, retries, and memory primitives. No LangChain, CrewAI, LlamaIndex, OpenAI Agents SDK, or Claude Agent SDK.

Amendment (2026-05-21). Pydantic-ai is now the provider/model abstraction layer (see ADR-006); app code talks to pydantic-ai’s Agent / Model interface, not the openai SDK directly. The agent harness (loop, tool dispatch, retries, memory) remains hand-rolled. Letta is permitted in specific levels where it meaningfully reduces code. See I7.

Consequences. Slower L6 ramp-up; more code in our repo; the resulting skill (“a small Claude Code on my laptop”) is more credible than a framework integration. The amendment leaves room for comparative experiments without abandoning the hand-rolled identity.

Alternatives
LangGraph (most popular); Pydantic-AI (newest credible); OpenAI Agents SDK (vendor-blessed)
Owner
Scott
ADR-003

FastAPI backend on every level; frontend varies

Context. The eventual goal is an Obsidian plugin. Obsidian plugins are TypeScript but consume HTTP services.

Decision. Every level exposes a FastAPI backend with a stable API contract. The frontend is per-level (CLI/TUI/web); an Obsidian plugin can be wired on top of any level.

Consequences. Slight overhead at L1; clean integration story; every level is consumable from the same Obsidian plugin once written.

Alternatives
CLI-only (no API); separate plugin per level
Owner
Scott
ADR-004

Anchor every level in building one growing personal assistant

Context. We need a unifying theme across ten levels that absorbs the breadth of the curriculum (chat, memory, skills, evals, tools, agents) without contortion. We considered three framings: general-purpose-each-app, PKM-anchored-vault, and conversation-anchored-assistant. The strongest analogs we studied (gptme, simonw/llm, Cline, Claude Code itself) are personal assistants that grew capabilities over time, not pipelines over corpora.

Decision. Every level contributes to one growing personal assistant. The assistant talks to the user (L1–L2), learns about the user (L3), develops skills for the user’s recurring tasks (L4), is graded on its work (L5), gains tools when the conversation demands them (L6–L7), proposes its own new skills (L8), takes on long-horizon work (L9), and improves itself over time (L10). The Obsidian vault is the memory substrate where everything the assistant knows lives. External surfaces (Bluesky, web, calendars) are tools the assistant earns, not the foundational substrate.

Consequences. Apps feel like one product family (the assistant grew). Each level is immediately personally useful (L1 alone is a chat tool you use daily; L4 is a personalized agent before any tool fires). External-integration risk is bounded: if Bluesky or any other tool turns out to be the wrong fit at L9, it gets swapped without disturbing L1–L8. The Obsidian-plugin SaaS direction stays credible: the assistant is exactly the kind of thing PKM users want as a plugin, but the assistant is the product, not the plugin.

Alternatives
General-purpose framing (each level is its own product); PKM-anchored (vault content is the source data); data-pipeline (ingest a corpus per level)
Owner
Scott
ADR-005

Skills-spec compliance from day one

Context. The agentskills.io spec is the cross-vendor convergence point (Claude Code, Codex CLI, OpenCode, Cursor). Anything we write to the vault as instructions for an agent has a canonical format.

Decision. Markdown files our agents write to the vault that contain agent-readable instructions follow the SKILL.md frontmatter spec. Our own agent reads them too.

Consequences. Same vault is consumable by our agent and by Claude Code/Cursor/Codex: free portfolio multiplier. Locks in some frontmatter shape early.

Alternatives
Bespoke frontmatter; no skills at all
Owner
Scott
ADR-006

Use pydantic-ai as the provider abstraction layer

Context. ADR-001 commits us to OpenAI through the build phase, and I7 commits us to hand-rolling the agent harness. We still want future provider expansion (Anthropic, Gemini, open-weight via Ollama) to be a config change, not a code change. Three shapes considered: a thin in-repo app.provider.LLM wrapper (more code, no community-tested provider parity); a gateway daemon like openclaw (multi-channel reach we do not need here, heavier ops surface); pydantic-ai (model-agnostic Agent/Model interface from the Pydantic team, lightweight, Pythonic, type-safe). Pydantic-ai is the shape we’d hand-roll anyway, and it’s already permitted by ADR-002’s amendment.

Decision. Use pydantic-ai as the provider/model layer. App code uses pydantic-ai’s Agent / Model types and Pydantic result types; it never imports openai directly. Provider swap is a model-string change in config (openai:gpt-4o-mini → anthropic:claude-haiku-4-5). The hand-rolled agent harness (loop, memory, skills, planner/worker per I7) sits on top of pydantic-ai’s per-call provider abstraction. Specific levels may opt to use more of pydantic-ai (e.g., its built-in tool dispatch) as deliberate detours per ADR-002’s amendment.

Consequences. One small framework dep (pydantic-ai) added at L1, targeted at exactly this seam. Type safety throughout (Pydantic-native return types). Provider portability becomes config. I7 stays intact for the agent harness; pydantic-ai’s role is the narrow provider/model layer, not the full agent loop. Embeddings route through the indexer’s own provider-agnostic interface (e.g., chromadb’s embedding functions), not through pydantic-ai.

Alternatives
Direct OpenAI SDK calls everywhere (prior draft); thin in-repo app.provider.LLM wrapper (more glue code, no community parity); litellm (procedural, less type-safe, ergonomically clunkier for structured outputs); gateway daemon like openclaw (multi-channel reach not needed here; could still be layered behind pydantic-ai later if a level wants it)
Owner
Scott
ADR-007

Multi-principal from day one (human user plus agent users)

Context. The assistant has at least two users: the author (human) and Claude Code (an agent acting on the author’s behalf). Multi-principal support is much cheaper to design in from L1 than to retrofit at L7. A single-principal design embeds “the user” everywhere (in session IDs, memory tables, trace partitions, recall corpora) and retrofitting requires a migration on every persisted artifact.

Decision. Every API call identifies its calling principal (header for HTTP, argument for CLI). Conversation sessions, semantic-recall corpora, declarative-memory user-facts, and trace records are partitioned by principal. Skills, shared project knowledge, and global facts are not partitioned. Authentication is simple from L1 (header-based API keys for agent principals; OS user for the local CLI) and can grow without schema change.

Consequences. An extra header/argument flows through every endpoint and persistence call from L1. Memory stores carry a principal column or directory partition. Claude Code can ask the assistant “what do you remember about Scott?” through the same API surface the human CLI uses; the assistant can hold distinct memories about Claude Code itself separately from memories about Scott.

Alternatives
Single-user with retrofit later (deferred cost much higher); multi-tenant with full RBAC (too heavy for the curriculum); per-process principal pinning (no programmatic principal switching, breaks the Claude-Code-as-second-user goal)
Owner
Scott

Cross-cutting patterns

Patterns that surfaced across the analog research and become load-bearing structures of the portfolio: designed once, reused across levels.

Memory-as-files

The dominant memory pattern for code agents in 2026 (cline Memory Bank, kepano skills, obsidian-mind). The vault is the memory. The assistant reads markdown on session start, writes markdown when it learns something, and amends markdown when facts change. No database needed for the long-term tier. Procedural memory equals a markdown playbook. Central to L3, L4, and L8.

Conversation-as-corpus

The assistant’s growing chat history is itself the retrievable corpus for L2. No external dataset needed; the substrate is whatever you’ve actually talked about. Same retrieval primitives (embeddings + BM25 + recency) that would be applied to external documents in the original PKM framing now apply to your own past exchanges.

Plan/build agent split

Two-role pattern in opencode (plan/build), Aider (architect/editor), SWE-agent. One agent reads-and-proposes (read-only safety), another applies edits. More reliable than multi-agent free-for-alls: Cognition’s Don’t Build Multi-Agents argues this directly. Used at L9.

Vault-write primitives (designed at L1, reused forever)

Three primitives lifted from Ar9av’s conventions, designed once in L1’s vault-write layer for session transcripts and reused for every subsequent kind of write (memory updates, skill drafts, dossier sections):

  • Manifest with SHA-256 content-hash delta tracking: skip re-write on hash match.
  • Staging directory + .patch.md files: agent proposes diffs against existing pages, never blind-overwrites.
  • Provenance markers (^[inferred], ^[ambiguous]) plus provenance: frontmatter ratios. These distinguish what the assistant knows directly from what it inferred.

Three-tier structured-output retry (kytmanov)

Tier 1: native structured outputs. Tier 2: regex extraction from a fenced block. Tier 3: retry with prior validation error embedded in the prompt. Treats the model as a self-correcting compiler. First used at L4 (validating skill frontmatter on load) and L6 (parsing tool-call JSON); reused wherever schema validation fails downstream.

Skills-spec progressive disclosure (Anthropic)

Three load levels: (1) name + description always in context (~100 tok/skill); (2) body loaded on match; (3) references/ and scripts/ loaded only when the workflow points to them. We mirror this for our own assistant’s skill catalog from L4 onward; agent-written skills at L8 follow the same shape.

Eval strategy: solving the no-SME problem

Without a domain expert to grade outputs, evaluations must be deterministic, judged-by-rubric, or trajectory-based.

Three reusable strategies the portfolio mixes per level:

  1. Verifiable domains. Tasks where correctness is mechanical: stream-cancel correctness, persistence round-trip, retrieval (labeled past-exchange found?), planted-fact recall, skill-match precision, schema validation. L1 persistence, L2 retrieval, L3 memory probes, L4 skill match, L9 task outcomes use this.
  2. LLM-as-judge with rubrics. Works when the rubric is tight enough that two judges agree. Validate by sampling and self-checking. L1 multi-turn coherence, L2 faithfulness, L4 skill-following adherence, L8 agent-drafted skill quality.
  3. Trajectory evals. Score the path, not just the answer: right tool, right order, right args, right number of iterations, unnecessary calls skipped. Catches regressions when outputs look fine. L6 onward.
Why L5 lives where it does

L5 (the eval harness) sits at the pivot because L1–L4 can be graded deterministically: the easy way to learn how evals work. By L6 the outputs are too open-ended for pure determinism, and we need the L5 muscle already trained. Building L5 after L4 is the cheapest possible eval education.

RiskSeverityLikelihoodMitigation
Scope creep: “production” quality gets confused with enterprise quality; levels drag on. High Likely Strict definition-of-done per level. Ship at “good enough for a public repo”, not “deployable to paying customers.”
L10 has no production analog and may become an indefinite research project. Medium Likely Time-box at two weeks. If the loop doesn’t close, ship as a writeup-heavy capstone rather than full code.
Eval costs balloon when L5 retro-applies to L1–L4. Medium Possible Cache LLM-as-judge calls aggressively (diskcache); use small models for judging where feasible.
Memory accumulation drift: after months of conversations the L3 declarative store goes stale or self-contradicts. Medium Likely L3 includes contradiction-on-update logic (newer facts override older with citation). L5 grades memory recency and consistency as part of routine eval runs.
User-as-only-tester limitation: the author is the only daily user, so eval coverage skews to one person’s patterns. Medium Possible L5 fixtures include synthesized chat sessions across distinct user personas. Invite at least one external user by L4 to widen the trace surface.
Markdown / vault format assumptions break (Obsidian Bases spec, frontmatter conventions, wikilink semantics). Low Possible Pin to agentskills.io spec where applicable. Vault is plain files; Obsidian-specific surface is a thin adapter, not core code.
Author drift: restart a level with the latest pattern after learning more, never finish. High Likely Each level is frozen at its commit. Apply learnings to the next level, not previous ones.
