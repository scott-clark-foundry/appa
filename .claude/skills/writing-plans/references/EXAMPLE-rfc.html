Feature design · RFC-014

Make the agent chat feel alive: stream tokens as they arrive

Replace request/response polling with Server-Sent Events so users see assistant tokens within ~120ms of generation. Adds visible motion to every reply, halves perceived latency, and unblocks tool-call rendering.

Intent

Goal

First visible token within 150ms of message submission (p50). Tool-call previews stream as they're decided. The chat surface conveys activity without requiring users to read a spinner.

Non-goals
  • Multiplayer / shared sessions: out of scope for this RFC.
  • Replacing the underlying provider client; we wrap, we don't fork.
  • Mobile-app parity: web only this cycle; native follows in sprint 25.
Open questions
  1. Do we reuse the existing /messages route with an Accept: text/event-stream header, or expose /messages/stream as a sibling? Leaning sibling for cache-layer clarity.
  2. How do we handle a client that disconnects mid-stream? Buffer + resume, or abandon the generation?
  3. Do tool-call results stream too, or do we wait for completion of each tool before streaming the next assistant message?

Context

Every interactive system that lasts more than 100ms invents a way to look busy. Ours invented a spinner. We can do better.

The current chat blocks on a single POST /messages that returns the full assistant turn when generation completes. Median time-to-first-byte is 2.4s; p95 is 7.1s. During that window the UI shows a pulsing dot. Session-replay analysis shows users re-reading their own prompt three to four times waiting for the dot to resolve.

Our provider already exposes a streaming endpoint. The blocking is on our end: the route handler awaits completion, the React client awaits the JSON body, and the worker awaits the database write. None of these are necessary.

Assumptions
  • Our CDN (Cloudflare) does not buffer text/event-stream responses. Confirmed via the /health/stream-probe endpoint last Thursday.
  • The agent runner can be made re-entrant within a single request without rewriting the tool-dispatch loop. Spike branch spike/streaming-runner proves this works for the simple-tool path.
  • ~1% of users are on networks that proxy and buffer SSE. We accept graceful degradation for them.

Design

Event schema

One envelope, six event kinds. Every event has id, seq, and t (server ms since stream open). The client treats unknown kinds as no-ops so we can extend without versioning.

events.ts typescript 
type StreamEvent =
  | { kind: 'token';       text: string }
  | { kind: 'tool-start';  toolId: string; name: string; input: unknown }
  | { kind: 'tool-result'; toolId: string; ok: boolean; output: unknown }
  | { kind: 'meta';        usage?: Usage; model?: string }
  | { kind: 'error';       code: string; recoverable: boolean; message: string }
  | { kind: 'done';        finishReason: 'stop' | 'length' | 'tool' | 'error' };

interface Envelope<K = StreamEvent> {
  id: string;   // ULID, monotonic
  seq: number;  // per-stream ordinal
  t: number;    // server ms since stream open
  event: K;
}

Client buffer

Tokens land in a ring buffer keyed by messageId; a 16ms rAF tick flushes them into the React tree as a single state update. This avoids the layout thrash we saw in the prototype when tokens arrived at >120/s.

Key insight

The perception of speed is owned by the client. The server can emit tokens at 200/s and feel slow if React re-renders the whole conversation per token; it can emit at 30/s and feel instant if we coalesce on a frame budget.

State machine

A streaming bubble has four states. The transitions matter more than the names.

                      ┌──────────┐
   ─── submit ───────▶│ pending  │
                      └────┬─────┘
                           │  first token / first tool-start
                           ▼
                      ┌──────────┐    tool-start    ┌─────────┐
                      │ streaming│ ───────────────▶ │ tool    │
                      └────┬─────┘                  └────┬────┘
                           │ done                        │ tool-result
                           ▼                             │
                      ┌──────────┐                       │
                      │ settled  │ ◀─────────────────────┘
                      └──────────┘

Decisions

ADR-014.1

SSE over WebSocket

Context. Both deliver progressive output. SSE is HTTP/1.1-compatible, plays nicely with our edge worker, and survives ad-blockers that aggressively close upgraded connections.

Decision. Server-Sent Events. Browsers handle reconnect automatically; we keep one request per turn.

Consequences. No client-to-server messaging mid-turn (we'd need a separate channel for cancellation; we'll use DELETE /messages/:id). We accept this.

Alternatives
WebSocket, long-poll, gRPC-web server streaming
Owner
r. okafor
ADR-014.2

Persist tokens server-side, never reconstruct from client

Context. Tempting to write only the final message and rebuild from client-streamed bytes on reload. This is fragile (clients lie, networks lose).

Decision. Token batches flush to durable storage every 200ms server-side. Client reload reads from durable storage, not memory.

Consequences. +~10% write IOPS during peak. Acceptable based on Postgres headroom (~6× current).

ADR-014.3

Don't ship without resume

Context. A user closing the tab mid-generation and reopening should see the message complete on its own. Without this, streaming is a downgrade for slow networks.

Decision. Block launch on a working GET /messages/:id/stream?from=:seq. Server replays buffered events.

Risks

RiskSeverityLikelihoodMitigation
Corporate proxies buffer SSE, breaking streaming for ~1% of users. Medium Certain (small cohort) Detect via heartbeat absence within 3s; fall back to single-response polling silently. Surface "slow network" hint after 5s.
React renders thrash at >100 tokens/sec. Low Likely without coalesce rAF tick + ring buffer (this RFC). Validated in spike at 240 tok/s.
Provider mid-stream errors leak through as broken UTF-8. High Rare but observed Decode at envelope boundary, not byte; emit error event and finalize bubble with partial content + retry affordance.
Durable store becomes hot key during bursts. Medium Possible at 10× load Per-message stream id is a ULID; writes spread naturally. Load test confirms behaviour at 5×; we accept watching at 10×.

Acceptance

  1. p50 time-to-first-token < 150ms; p95 < 400ms. qa obs
  2. Killing the browser tab mid-stream and reopening within 60s resumes the message to completion. qa manual
  3. Tool calls render tool-start < 300ms after model decision. qa obs
  4. No regression in chat-completion error rate (currently 0.4%). qa obs
  5. Lighthouse interaction-to-next-paint on chat page ≤ 200ms. qa obs
