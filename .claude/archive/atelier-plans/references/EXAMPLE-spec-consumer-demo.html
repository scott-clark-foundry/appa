Context

The repo whose job is to demonstrate the scaffold's override surface currently shows the override surface from last slice.

The skills-and-tools slice (2026-05-15-skills-and-tools-design.md) shipped three seams: ToolCapability untrusted-content wrapping, SkillRegistry discovery, and the enum-constrained Skill tool: and merged to main. The reference consumer at ~/eval-agent-2-consumer still exercises only the run-row refactor: it overrides the data directory, injects one contributor, and reuses the scaffold's REPL. The headline features of the most recent slice are invisible in the one repo whose job is to demonstrate the scaffold's override surface. A reviewer opening the consumer sees a demo that is one slice stale.

Two concrete gaps

1. No consumer tools surface. create_agent() exposes no surface for a consumer to register its own function tools: build_tools(registry) is called internally and returns only the Skill tool. The J-skills-02 journey reached the <untrusted_content> path only by monkeypatching build_tools; that journey's Pre-flight deliberately declined a test_tools= parameter as "production API surface for a test-only need."

2. Skills directory resolution is wrong for consumers. default_skill_dirs() resolves the project root from the scaffold package's __file__; the consumer installs the scaffold as a non-editable uv path dependency, so that path points into site-packages and never at the consumer's own skills. Even with an editable install it would be wrong here: the skills the consumer wants to demonstrate live in the consumer repo.

This spec defines the slice that closes both gaps: a small scaffold change adding a validated tools= parameter to create_agent(), and consumer changes that author one skill, register one untrusted tool through the new parameter, and document both override surfaces. The tools= parameter is not the test-only need the J-skills-02 journey declined: it is a genuine consumer API, which is why it belongs on the public factory.

Goals

  1. G1
    create_agent() accepts tools: list[Tool] | None = None; consumer-supplied tools are combined with the scaffold's own (build_tools(registry)), and the combined set is passed through the existing validate_tool_metadata(): verified by a scaffold unit test that registers a consumer tool and asserts it reaches the agent's tool set.
  2. G2
    A consumer tool registered via tools= missing trust_class or side_effect_class causes the same fail-fast ValueError at agent construction as a scaffold tool would: verified by a scaffold unit test (extends invariant I1 of the skills-and-tools spec to consumer tools).
  3. G3
    A consumer tool with trust_class="untrusted" has its result wrapped in <untrusted_content> … </untrusted_content> by the already-installed ToolCapability, with no ToolCapability change: verified by a scaffold unit test over a consumer-shaped untrusted tool.
  4. G4
    The consumer ships one self-contained SKILL.md under its own .agents/skills/, and create_agent() is called with an explicit skill_dirs rooted at the consumer project (resolved from the consumer package location, not the current working directory) so the authored skill appears in Run.instructions and is activatable via Skill(name).
  5. G5
    The consumer registers one trust_class="untrusted", side_effect_class="read" local-file-reader tool that reads a file shipped under data/untrusted/; that file contains a planted, benign mock prompt-injection line so the demo visibly shows the line reaching the model inside <untrusted_content> markers.
  6. G6
    The consumer README.md documents the two new override surfaces (skill_dirs, tools=); the stale "register a tool on core_agent directly or fork the factory" advice is removed.
  7. G7
    Journey J-consumer-skills-tools runs end-to-end against the live consumer and passes all checklist steps.
  8. G8
    A consumer tool whose name collides with a scaffold-authored tool (e.g. a tool named Skill) causes a fail-fast ValueError at agent construction, naming both tools: create_agent() checks the combined scaffold+consumer set for duplicate names so a consumer tool cannot silently shadow a scaffold tool: verified by a scaffold unit test.

Non-goals

  1. NG1
    Demonstrating the L1 side-effect gate (external / irreversible routing): the demo's untrusted tool is side_effect_class="read", and the gate is a deliberate L1 no-op anyway. Deferred to whenever an L3 approval queue lands.
  2. NG2
    Tier-3 skill resources (scripts/, references/, assets/): the consumer's skill is tier-1/tier-2 only. The consumer's existing .claude/skills/diagram-expert is a Claude Code developer skill, not agent-facing, and is untouched.
  3. NG3
    A new scaffold tool subsystem or any production tool shipped by the scaffold: the scaffold still authors only the Skill tool. This slice adds a registration surface, not tools.
  4. NG4
    Changes to skill discovery, frontmatter parsing, catalog rendering, or the Skill tool: those are unchanged from the skills-and-tools slice.
  5. NG5
    A test suite in the consumer repo: the consumer stays "tiny on purpose." Consumer behavior is verified by the journey; the tools= parameter is unit-tested in the scaffold suite.
  6. NG6
    Discovering user-scoped skills (~/.agents/skills) in the consumer: the consumer passes only its own project skills directory so the demo and its journey are reproducible regardless of what the operator has installed. Adding the home directory is noted as a documented extension, not built here.

Design

Scaffold change: the tools= parameter

create_agent() gains one optional parameter:

scaffold/factory.pypython
def create_agent(
    key: ConversationKey,
    history_store: HistoryStore | None = None,
    config: AgentConfig = DEFAULT_CONFIG,
    session_id: str = "local",
    contributors: list[tuple[str, Callable[..., str]]] | None = None,
    skill_dirs: list[Path] | None = None,
    tools: list[Tool] | None = None,   # NEW: consumer-supplied function tools
) -> PersonalAgent: ...

tools accepts pydantic_ai.tools.Tool instances, not raw callables: the trust_class / side_effect_class metadata convention can only ride a Tool's metadata dict, and the existing validate_tool_metadata() reads .metadata. A raw callable passed here carries no metadata and is therefore rejected by that validator with the existing fail-fast error: acceptable, but the parameter's type and docstring state Tool explicitly.

The factory combines the scaffold's own tools with the consumer's, validates the union, and passes the union to PydanticAgent. Today's body is:

tools = build_tools(registry)
validate_tool_metadata(tools)
core_agent = PydanticAgent(..., tools=tools, ...)

After this change the validated set is build_tools(registry) plus the consumer list. build_tools() itself is unchanged: it still owns only the Skill tool. Tool ordering within the combined set is not load-bearing.

create_agent() also checks that the combined set has no duplicate tool names, raising the same kind of fail-fast ValueError as validate_tool_metadata(): naming the colliding consumer tool and the scaffold tool it would shadow. This is a deliberate scaffold-owned check, not a delegation to Pydantic AI.

Why scaffold-owned

A consumer tool whose name collides with a scaffold tool (Skill today, a MemoryStore tool later) could otherwise silently shadow it, and the scaffold's Skill tool is trusted while a consumer's same-named replacement need not be. Silent shadowing of a trusted built-in is the same security hole the metadata fail-fast exists to prevent, so the failure surface is made identical: deterministic, at construction, naming the offender. Checking the union for duplicates (rather than a hard-coded scaffold name list) covers future scaffold tools automatically.

Consequences fall out of existing architecture

  • Validation: because the combined set flows through validate_tool_metadata(), a consumer tool missing either metadata field fails fast at construction, exactly as invariant I1 requires for scaffold tools. The guarantee is now uniform across scaffold and consumer tools.
  • Untrusted wrapping: ToolCapability is registered on the agent via build_capabilities() and reads trust_class from each tool's ToolDefinition.metadata at execution time (after_tool_execute). A consumer tool marked trust_class="untrusted" is wrapped with no ToolCapability change.
  • Side-effect gating: likewise handled by before_tool_execute for any consumer tool whose side_effect_class is external / irreversible. Not exercised by this demo (NG1), but the parameter does not special-case it.

Consumer change: the demonstration skill

The consumer authors one SKILL.md under <consumer-project>/.agents/skills/<skill-name>/SKILL.md. The skill is a small, self-contained behavior skill: its body is a short procedure or style directive the agent adopts when activated: and is deliberately unrelated to the untrusted tool, so skill discovery/activation and untrusted-content wrapping read as two independent demos.

The frontmatter follows the contract the SkillRegistry already enforces: name (lowercase letters and hyphens, ≤64 chars, matching the parent directory name) and description (≤1024 chars). The exact subject and prose of the skill are the implementer's to choose; the journey asserts only that the skill is catalogued in Run.instructions and that its frontmatter-stripped body is returned inside the Skill activation wrapper.

create_agent() is called with an explicit skill_dirs pointing at the consumer's .agents/skills directory. The path must be resolved from the consumer package's own location (e.g. relative to __file__), not from the current working directory, so the demo works regardless of where personal-agent is launched. Only the consumer's own skills directory is passed (NG6).

Consumer change: the untrusted local-file-reader tool

The consumer defines one function tool, wrapped as a pydantic_ai.tools.Tool, with:

  • metadata={"trust_class": "untrusted", "side_effect_class": "read"}
  • a single string parameter naming a note to read
  • a body that reads a file from a fixed data/untrusted/ directory and returns its text; that directory MUST be resolved from the consumer package's own location, not from the current working directory

The pattern looks like:

consumer/tools/read_note.pypython
def read_note(name: str) -> str:
    """Read a saved note by name. Returns the note's text."""
    path = UNTRUSTED_DIR / name
    return path.read_text(encoding="utf-8")

read_note_tool = Tool(
    read_note,
    metadata={"trust_class": "untrusted", "side_effect_class": "read"},
)

The repo ships at least one file under data/untrusted/ whose content includes a benign, clearly-mock prompt-injection line (for example, a line instructing the agent to ignore prior instructions). The threat the demo illustrates is real: file content is outside the program's control: while the payload itself does nothing harmful. The journey asserts that this planted line reaches the model enclosed in <untrusted_content> markers, demonstrating that the agent receives it as data, not as instructions.

Out of scope

File-path handling (rejecting traversal outside data/untrusted/, behavior on a missing file) is the implementer's to decide; the demo's job is to show the wrapping, not to be a hardened file server.

Consumer change: documentation

The consumer README.md is updated so its "What this consumer overrides" table and "How to extend further" section reflect reality:

  • Add rows/notes for skill_dirs (authoring a skill, passing an explicit project-rooted directory) and tools= (registering a consumer Tool with required metadata).
  • Remove the stale "Add a tool" guidance that tells consumers to register on core_agent directly or fork the factory: that path bypassed validate_tool_metadata() and is superseded by tools=.

Invariants

  1. I1
    (consumer-extended) Every function tool reaching a model request: scaffold-authored or consumer-supplied via tools=: has both trust_class and side_effect_class in its ToolDefinition.metadata, enforced by validate_tool_metadata() at construction. (This is invariant I1 of the skills-and-tools spec, now covering consumer tools.)
  2. I2
    The consumer's skill is discovered only from the explicit project-rooted skill_dirs it passes: discovery does not depend on the install mode of the scaffold dependency or on the process's working directory.
  3. I3
    No two tools in the set create_agent() passes to the agent share a name: a consumer tool supplied via tools= cannot shadow a scaffold-authored tool. A collision fails fast at construction with a ValueError naming both tools.

Journey

J-consumer-skills-tools

Reference consumer demonstrates two independent override surfaces

Pre-conditions

  • The consumer repo has one SKILL.md under .agents/skills/ and one file under data/untrusted/ containing a planted mock prompt-injection line.
  • OPENAI_API_KEY is available.

Steps

  1. Start the consumer; the skills catalog in Run.instructions lists the authored skill by name and description.
  2. The agent activates the skill via Skill(name); the tool result contains the frontmatter-stripped skill body inside the <skill name="..."> wrapper.
  3. The agent invokes the consumer's untrusted file-reader tool.
  4. The untrusted tool's result reaching the model is enclosed in <untrusted_content> … </untrusted_content> markers, and the planted injection line appears inside those markers.

This journey lives in the scaffold repo's docs/journeys/, following the precedent of docs/journeys/2026-05-13-run-row-refactor-checkpoint.md, which already documents consumer behavior from the scaffold repo. Keeping it there avoids introducing a docs/journeys pipeline into the deliberately tiny consumer.

Testing strategy

  • Unit-tested: create_agent(tools=...) registration: pass a consumer Tool with valid metadata; assert it appears in the constructed agent's tool set alongside the Skill tool.
  • Unit-tested: consumer-tool metadata fail-fast: pass a consumer tool missing a metadata field; assert create_agent() raises ValueError at construction.
  • Unit-tested: consumer-tool name-collision fail-fast: pass a consumer tool named Skill; assert create_agent() raises ValueError at construction naming both tools.
  • Unit-tested: consumer untrusted tool wrapping: register a consumer-shaped trust_class="untrusted" tool, exercise ToolCapability.after_tool_execute, assert the result is enclosed in <untrusted_content> markers. (May reuse the existing fixture tools in app/tests/fixtures/tools.py.)
  • Journey-tested: the end-to-end consumer demo: J-consumer-skills-tools runs against the live consumer and passes every step.
  • Not tested: the L1 side-effect gate: the demo's untrusted tool is read; the gate is a deliberate L1 no-op (NG1).
  • Not tested: consumer internals beyond the journey: the consumer keeps no test suite (NG5).

Forward-compatibility

  • The tools= parameter is the surface every future consumer tool flows through. When a MemoryStore or other tool-bearing subsystem lands, scaffold-authored tools still come from build_tools(); nothing about the consumer-tool path changes.
  • The demo's untrusted tool is side_effect_class="read". If an L3 approval queue replaces the L1 gate, a consumer tool marked external / irreversible would route through it automatically: no change to the tools= contract.
  • The consumer's explicit skill_dirs insulates it from the default_skill_dirs() editable-install assumption. If the scaffold later resolves project skills differently, the consumer is unaffected because it never relies on the default.
