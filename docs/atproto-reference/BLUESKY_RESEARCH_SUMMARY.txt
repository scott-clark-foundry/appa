KEY FINDINGS:
================================================================================

THREAD DATA STRUCTURES:
- Single post type: app.bsky.feed.post
- Three thread node types: ThreadViewPost, NotFoundPost, BlockedPost
- Posts can be: replies (threaded), quotes (embedded), or standalone
- Replies require both parent AND root references
- Quotes use embed field, not reply field
- Five embed types: images, video, external, record, recordWithMedia

API RESPONSE FORMAT:
- Endpoint: GET https://public.api.bsky.app/xrpc/app.bsky.feed.getPostThread
- Parameters: uri (required), depth (0-1000, default 6), parentHeight (0-1000, default 80)
- Response: { thread: (ThreadViewPost|NotFoundPost|BlockedPost), threadgate: ? }
- Parent chain: nested tree (parent.parent.parent...)
- Replies: flat array of direct children only

THREAD HIERARCHY EXAMPLES:
================================================================================

SIMPLE REPLY THREAD:
  Root Post
    └── Reply 1
    └── Reply 2
    └── Reply 3

NESTED REPLY (from API perspective):
  [Ancestor 1]
    └── [Ancestor 2] <- parent field (nested)
        └── Root Post <- thread object
            ├── Reply 1 <- replies[0]
            ├── Reply 2 <- replies[1]
            └── Reply 3 <- replies[2]

REPLY VS QUOTE DISTINCTION:
  Reply post structure:      { record: { reply: { parent: {...}, root: {...} } } }
  Quote post structure:      { record: { embed: { $type: "app.bsky.embed.record" } } }

DIFFERENT POST STATES IN THREAD:
  ThreadViewPost: Normal, accessible post with full content
  NotFoundPost: Deleted or unavailable (URI only)
  BlockedPost: Blocked by user or user is blocked (URI + metadata)

DATA EXTRACTION PATTERNS:
================================================================================

BASIC TRAVERSAL:
- Access root: response.thread.post
- Access ancestors: response.thread.parent (iterate upward)
- Access replies: response.thread.replies[] (iterate direct children)

TYPE DISCRIMINATION:
- isinstance(node.thread, models.AppBskyFeedDefs.ThreadViewPost)
- isinstance(node.thread, models.AppBskyFeedDefs.NotFoundPost)
- isinstance(node.thread, models.AppBskyFeedDefs.BlockedPost)

EMBED IDENTIFICATION:
- models.AppBskyEmbedImages.Main
- models.AppBskyEmbedVideo.Main
- models.AppBskyEmbedExternal.Main
- models.AppBskyEmbedRecord.Main (quote post)
- models.AppBskyEmbedRecordWithMedia.Main (quote + media)

REPLY DETECTION:
- post.record.reply is not None → it's a reply
- post.record.reply.parent.uri → immediate parent
- post.record.reply.root.uri → thread root

QUOTE DETECTION:
- isinstance(post.record.embed, models.AppBskyEmbedRecord.Main) → quote post

TROUBLESHOOTING GUIDE:
================================================================================

Q: Import error for atproto?
A: pip install --upgrade atproto

Q: Type hints not working?
A: Ensure using Python 3.9+ with proper IDE support

Q: Rate limited (429 responses)?
A: Implement exponential backoff (see Example 5)

Q: Models missing attributes?
A: Check atproto version, may need upgrade

Q: Thread structure unexpected?
A: Verify parent is nested, replies are flat array

Q: Post type detection failing?
A: Use isinstance() checks, not string comparisons

Q: Embed information incomplete?
A: Some fields may be None, check for optional values


PERFORMANCE NOTES:
================================================================================

API CALL COST:
- Single getPostThread call retrieves full thread
- Typical response time: 200-500ms
- Respects rate limits (see Bluesky docs for limits)
- No need for multiple calls to build tree

MEMORY USAGE:
- Pydantic models are lightweight
- Large threads (1000+ posts): minimal overhead
- Recursive traversal: O(n) complexity

OPTIMIZATION TIPS:
1. Use depth/parentHeight wisely (0-100 usually sufficient)
2. Cache thread responses when possible
3. Batch multiple thread requests
4. Consider async client for concurrent requests
